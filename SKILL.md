# 会议录音总结

## 启动

1. 确认运行目录可持久化，且已安装 Python 3.11 和 `ffmpeg`。
2. 按 `references/hermes-deployment.md` 设置转写与总结模型环境变量。
3. 运行：

```bash
python3 scripts/server.py --host 0.0.0.0 --port 8765
```

4. 将 Hermes 暴露的 HTTPS 地址发送给用户。手机浏览器只有在 HTTPS 或 localhost 下才能调用麦克风。

## 工作流

1. 创建会议，至少记录会议名称；鼓励补充主持人、参会人员、关联项目和会议目标。
2. 用户确认参会者知晓录音后开始录音。浏览器每 30 秒上传一个片段，服务端立即落盘并登记 SQLite。
3. 暂停时停止本地采集；继续时保持原会议 ID 并追加片段。结束前检查待上传片段为零。
4. 点击“结束并生成纪要”，服务端合并片段、调用转写服务、再调用总结模型生成 JSON 结构。
5. 展示完整逐字稿与纪要草稿。先审核待办任务，再审核摘要和讨论内容。每项任务只展示业务类型、待办事项、负责人、具体要求和完成时间，适配手机微信快速查看；缺失值写“待确认”。
6. 由主持人编辑并点击“确认归档”。只有确认后的版本才是正式纪要。

## 强制输出结构

按 `references/minutes-schema.md` 生成，顺序固定为：会议信息、待办任务总览、待办任务明细、会议决策、会议目标、内容摘要、主要讨论事项、风险及阻塞事项、待确认事项、下次会议安排。待办任务是纪要核心，不得被摘要替代。

## 边界

- 不虚构发言、决策、负责人、截止时间或交付物。
- 不把“继续跟进、尽快处理、后续沟通”等模糊表述直接当成合格任务；必须在“具体要求”或“待确认事项”指出缺失的信息。
- 模型推断的信息必须进入“待确认事项”，不能混入已确认事实。
- 转写或总结失败时保留录音片段、会议元数据和已有逐字稿，允许重新处理。
- 原始录音默认本地保存；删除或调整保留期限必须由有权限的用户操作。
- 不在源码、SKILL.md、数据库或日志中写入 API 密钥。
